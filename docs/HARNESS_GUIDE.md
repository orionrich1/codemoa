# CodeMoa 하네스 운영 가이드

> **이 문서의 목적**: AI(Cursor)에게 일을 시키는 방법과, 각 역할이 해선 안 되는 것을 한 곳에서 확인하기 위한 운영 지침서.
>
> **하네스 수정 권한**: 테스터 + Owner 전용. 다른 역할은 수정 불가.

---

## 전체 구조

```
전역 방향 (.cursor/rules/00-global-direction.mdc)   ← 항상 적용
│
├── 기획자 하네스 (.cursor/rules/01-planner-harness.mdc)
├── 개발자 하네스 (.cursor/rules/02-developer-harness.mdc)
├── 디자이너 하네스 (.cursor/rules/03-designer-harness.mdc)
└── 테스터 하네스 (.cursor/rules/04-tester-harness.mdc)
```

하네스는 **Cursor가 항상 읽는 규칙 파일**이다. 역할을 지정하면 해당 하네스의 규칙으로 동작한다.

---

## 전체 흐름 (Workflow)

```
1. 기획자  →  docs/plan.md(통합) 또는 docs/[도메인]-spec.md 에 명세 반영
       ↓
2. 개발자  →  해당 md 를 읽고 코드 구현
       ↓
3. 디자이너 →  동일 기획서의 화면 목록을 읽고 템플릿 구현
       ↓
4. 테스터  →  구현된 코드가 하네스 규칙을 지켰는지 검토 + 보고
```

## AI에게 일 시키는 방법 (역할별 프롬프트)

### 기획자로 일 시키기

```
기획자로서 [기능명] 기획서를 작성해줘.

배경: [왜 이 기능이 필요한지]
사용자: [누가 사용하는지]
핵심 요구사항: [반드시 포함되어야 할 내용]
```

**출력물**: `docs/plan.md`(Part 추가) 또는 `docs/[도메인]-spec.md` **하나**에 반영 (동일 주제 복수 md 금지 — `01-planner-harness` 참고)

**예시**:
```
기획자로서 팀 모집 게시글 작성 기능 기획서를 작성해줘.
배경: 개발자들이 공모전/프로젝트 팀원을 구하는 게시판이 필요함.
사용자: 로그인한 회원 누구나.
핵심 요구사항: 모집 인원, 기술 스택, 모집 기간을 입력할 수 있어야 함.
```

---

### 개발자로 일 시키기

```
개발자로서 @[기획서 파일명] 을 구현해줘.
```

기획서 파일을 `@파일명` 으로 첨부하면 AI가 해당 명세를 읽고 구현한다.

**예시**:
```
개발자로서 @docs/team-recruit-spec.md 를 구현해줘.
```

```
개발자로서 @docs/team-recruit-spec.md 중 "게시글 목록 조회" 기능만 먼저 구현해줘.
```

**출력물**: Java 소스 코드 (Controller / Service / Entity / Repository / DTO)

**완료 시**: Owner가 **새로고침**으로 바로 확인할 수 있게 `bootRun` 재시작 등 필요한 조치 + 확인 URL 안내 (`00-global-direction`, `02-developer-harness` 참고).

---

### 디자이너로 일 시키기

```
디자이너로서 @[기획서 파일명] 의 [화면명] 화면을 구현해줘.
```

**예시**:
```
디자이너로서 @docs/team-recruit-spec.md 의 "팀 모집 목록" 화면을 구현해줘.
```

```
디자이너로서 @docs/team-recruit-spec.md 의 모든 화면을 구현해줘.
```

**출력물**: `src/main/resources/templates/views/[도메인]/` 하위 HTML 파일

**완료 시**: 템플릿/CSS 반영을 위해 서버 재시작이 필요하면 수행·안내하고, **새로고침(필요 시 강력 새로고침)** 으로 화면을 볼 수 있게 확인 URL을 남긴다 (`03-designer-harness` 참고).

---

### 테스터로 일 시키기

```
테스터로서 [도메인명] 도메인의 코드 품질을 검토해줘.
```

```
테스터로서 [파일명] 이 하네스 규칙을 준수하는지 검토해줘.
```

```
테스터로서 [규칙 내용] 을 하네스에 추가하는 것을 제안해줘.  (Owner 승인 후 적용)
```

**출력물**: 미해결 위반이 있을 때만 `docs/`에 품질 보고 md (위반 목록 + 수정 방향).

**통과 판정 시**: 미해결 위반·미충족 수락 조건이 **없다**고 판단하면, 테스터가 작성한 **해당 품질 보고 md만 삭제**한다. 기획서·DESIGN 등 다른 md는 삭제하지 않는다 (`04-tester-harness` 참고).

**주의 — `gradle test`·「테스트 다시 해줘」만으로는 코드 수정 불가**  
테스터 역할이어도, 위 문구만 있으면 **실행 결과·실패 요약·수정 방향 보고**까지만 한다.  
`src/main/**`, `build.gradle`, 테스트 통과용 `src/test/**` 수정(예: `@ArchIgnore` 남발)은 **개발자로서** 명시할 때 개발자 영역이다. 상세는 `04-tester-harness.mdc`의 **「역할 경계 — 자동화 테스트·빌드」** 참고.

## 전체 금지 목록 한눈에 보기

> 각 역할이 **절대 해선 안 되는 것**을 역할별로 모아놓은 참조 테이블.

### 기획자 금지

| # | 금지 내용 | 이유 |
|---|----------|------|
| 1 | 코드 작성 | 개발자 역할의 영역 |
| 2 | 구현 방법 지시 (어떤 기술을 쓸지 강제) | 개발자 자율성 침해 |
| 3 | 디자인 결정 (색상, 레이아웃) | 디자이너 역할의 영역 |
| 4 | 모호한 수락 조건 ("잘 동작해야 함") | 검증 불가능 |
| 5 | 하네스 파일 수정 | 테스터/Owner 전용 |

### 개발자 금지

| # | 금지 내용 | 이유 |
|---|----------|------|
| 1 | Entity에 `@Setter` 사용 | 불변성 위반, 의도 불명 상태 변경 |
| 2 | `FetchType.EAGER` 사용 | N+1 문제, 성능 저하 |
| 3 | Controller에서 Repository 직접 접근 | 레이어 의존 방향 위반 |
| 4 | Entity를 API 응답/View에 직접 반환 | 도메인 노출, 결합도 상승 |
| 5 | 기획서 요구사항 임의 변경·축소 | 기획자 역할의 결정사항 침해 |
| 6 | `System.out.println` 사용 | 운영 환경 로그 오염 |
| 7 | `@Lazy`로 순환 참조 회피 | 설계 문제 은폐 |
| 8 | 하네스 파일 수정 | 테스터/Owner 전용 |

### 디자이너 금지

| # | 금지 내용 | 이유 |
|---|----------|------|
| 1 | 인라인 스타일 (`style="..."`) 사용 | 유지보수 불가, 일관성 파괴 |
| 2 | 색상값 하드코딩 (`#3B82F6` 등) | CSS 변수 체계 파괴 |
| 3 | 아이콘 라이브러리 혼용 | 시각적 일관성 파괴 |
| 4 | 테이블에 `table-responsive` 누락 | 모바일 레이아웃 파괴 |
| 5 | 삭제 버튼에 확인 모달 없이 연결 | 실수 삭제 UX 문제 |
| 6 | Java/Service/Controller 코드 수정 | 개발자 역할의 영역 |
| 7 | 하네스 파일 수정 | 테스터/Owner 전용 |

### 테스터 금지

| # | 금지 내용 | 이유 |
|---|----------|------|
| 1 | 프로덕션 코드(`src/main/**`) 직접 수정 | 발견·보고만 담당 |
| 2 | `build.gradle` 등 빌드·의존성 스크립트 수정 | 개발자/Owner 영역 |
| 3 | 테스트 녹색화를 위한 `src/test/**` 임의 수정 (`@ArchIgnore`·슬라이스 우회 등) | 규칙 무력화·역할 혼선 |
| 4 | Owner 승인 없는 하네스 수정 | 하네스는 협의 사항 |
| 5 | 기능 추가·변경 제안 | 기획자 역할의 영역 |
| 6 | 위반 미해결 코드에 배포 허용 | 품질 게이트 역할 |
| 7 | 「테스트 실행」만 요청된 상황에서 임의로 구현·리팩터 | 역할 미전환 시 개발 행위 금지 |

## 하네스 수정 방법 (테스터 + Owner 전용)

### 수정 대상 파일
```
.cursor/rules/
├── 00-global-direction.mdc   ← 전역 규칙
├── 01-planner-harness.mdc    ← 기획자 하네스
├── 02-developer-harness.mdc  ← 개발자 하네스
├── 03-designer-harness.mdc   ← 디자이너 하네스
└── 04-tester-harness.mdc     ← 테스터 하네스
.cursorrules                  ← 프로젝트 기술 규칙
```

### 수정 절차
1. **테스터가 개선 제안** → Owner에게 보고
2. **Owner 승인** 후 테스터가 해당 `.mdc` 파일 수정
3. **변경 내용 공유** → 전 팀원에게 변경 사항 전달

### AI에게 하네스 수정 요청

```
테스터로서 개발자 하네스에 다음 규칙을 추가해줘:
"금지: xxx는 하지 않는다. 이유: yyy"
```

## 기획·명세 파일 관리 (`docs/`)

**고정(삭제·통합 대상 아님)**

- `HARNESS_GUIDE.md` — 본 가이드
- `DESIGN.md` — 디자인 시스템

**통합 기획(단일 파일)**

- `plan.md` — 전역 PRD·우선순위(Part A) + **메인 페이지**(Part B) + **AI 코딩 문제**(Part C) + **공통 백로그**(Part D)

**도메인별 단일 명세(주제당 1파일)**

- `community-spec.md` — 커뮤니티 게시판 UX·수락 조건 + 리치 HTML 정책(부록)
- `information-spec.md` — IT 정보(강좌·도서·공모전) 개선 기획
- `employment-team-spec.md` — 채용·팀 모집

**임시(테스터만, 통과 시 삭제)**

- `[주제]-quality.md` 등 품질 검토 보고 — 미해결 위반 있을 때만 존재

**원칙**

- 같은 주제에 `*-improvement.md`, `*-policy.md`, `main-page-plan.md` 등 **파일을 쪼개 새로 만들지 않는다.** 정책·개선 이력은 해당 `*-spec.md` 또는 `plan.md` 안의 섹션·부록으로 합친다.
- **미착수·선택 과제**는 `plan.md` **Part D**에만 적고, 도메인 `*-spec.md`에는 **동일 `[ ]`를 반복하지 않는다** (기획서 수명 주기).
- 기획서(해당 Part 또는 `*-spec.md`)가 없으면 개발을 시작하지 않는다.

## 기획서 수명 주기 (찌꺼기 방지)

**의도**: 개발·테스트 후에도 `*-spec.md` / `plan.md`에 **끝난 일인데 `[ ]`만 남거나**, **이미 해소된 진단 표**가 쌓이지 않게 한다. **운영 가이드·디자인**(`HARNESS_GUIDE`, `DESIGN`)은 예외로 항상 유지한다.

| 단계 | 담당 | 할 일 |
|------|------|--------|
| 구현 완료 | 개발자 | 이번 스코프 수락 조건에 대해 기획서 **`[x]` 갱신**을 기획자와 맞추거나, 완료 메시지에 **갱신할 파일·항목**을 명시. 미착수는 **축소가 아니라** 기획자에게 **`plan.md` 백로그 이전** 요청. |
| 명세 정리 | 기획자 | 완료 항목 `[x]`; 미착수는 **`plan.md`에만** 남기고 도메인 spec에서는 **해당 줄·블록 삭제** 또는 `→ 백로그: plan.md §…` 한 줄로 치환. **해소된 이슈 표**는 삭제 또는 짧은 히스토리만. |
| 통과 판정 | 테스터 | 이번 스코프가 “완료”인데 대응 명세에 **남은 `[ ]`** 가 있으면 통과로 보지 않고, 정리 후 재검토 권고(상세는 `.cursor/rules/04-tester-harness.mdc` 항목 10). |

규칙 전문: `.cursor/rules/00-global-direction.mdc`(요지), `01-planner-harness.mdc`, `02-developer-harness.mdc`, `04-tester-harness.mdc` 의 **「기획서 수명 주기」** 절.

## 자주 묻는 질문

**Q. 역할을 지정하지 않으면 어떻게 되나요?**  
A. 개발자 하네스가 기본으로 적용됩니다.

**Q. 기획서 없이 개발 요청을 해도 되나요?**  
A. 기획서 없이 시작하면 요구사항이 불명확해 재작업이 발생합니다. 반드시 기획자 역할로 기획서를 먼저 만드세요.

**Q. 하네스 규칙이 현실에 맞지 않아요.**  
A. 테스터에게 개선 제안을 요청하세요. 테스터가 Owner와 협의 후 수정합니다.

**Q. 디자이너가 백엔드 데이터 구조를 알아야 하나요?**  
A. 기획서의 "데이터 요구사항" 섹션을 참고하면 됩니다. 직접 Java 코드를 볼 필요는 없습니다.

**Q. 테스트 코드도 개발자 하네스를 따르나요?**  
A. 예. 테스트 코드 작성도 개발자 역할입니다. `src/test/` 하위 파일 작성 시 개발자 하네스를 따릅니다.
