# CodeMoa 하네스 운영 가이드

> **이 문서의 목적**: AI(Cursor)에게 일을 시키는 방법과, 각 역할이 해선 안 되는 것을 한 곳에서 확인하기 위한 운영 지침서.
>
> **하네스 수정 권한**: 테스터 + Owner 전용. 다른 역할은 수정 불가.

---

## 전체 구조

```
전역 방향 (.cursor/rules/00-global-direction.mdc)   ← 항상 적용
│
├── 기획자 하네스 (.cursor/rules/01-planner-harness.mdc)
├── 개발자 하네스 (.cursor/rules/02-developer-harness.mdc)
├── 디자이너 하네스 (.cursor/rules/03-designer-harness.mdc)
└── 테스터 하네스 (.cursor/rules/04-tester-harness.mdc)
```

하네스는 **Cursor가 항상 읽는 규칙 파일**이다. 역할을 지정하면 해당 하네스의 규칙으로 동작한다.

---

## 전체 흐름 (Workflow)

```
1. 기획자  →  [기능명]-spec.md 작성
       ↓
2. 개발자  →  [기능명]-spec.md 를 읽고 코드 구현
       ↓
3. 디자이너 →  [기능명]-spec.md 의 화면 목록을 읽고 템플릿 구현
       ↓
4. 테스터  →  구현된 코드가 하네스 규칙을 지켰는지 검토 + 보고
```

---

## AI에게 일 시키는 방법 (역할별 프롬프트)

### 기획자로 일 시키기

```
기획자로서 [기능명] 기획서를 작성해줘.

배경: [왜 이 기능이 필요한지]
사용자: [누가 사용하는지]
핵심 요구사항: [반드시 포함되어야 할 내용]
```

**출력물**: `docs/[기능명]-spec.md` 형태의 마크다운 기획서

**예시**:
```
기획자로서 팀 모집 게시글 작성 기능 기획서를 작성해줘.
배경: 개발자들이 공모전/프로젝트 팀원을 구하는 게시판이 필요함.
사용자: 로그인한 회원 누구나.
핵심 요구사항: 모집 인원, 기술 스택, 모집 기간을 입력할 수 있어야 함.
```

---

### 개발자로 일 시키기

```
개발자로서 @[기획서 파일명] 을 구현해줘.
```

기획서 파일을 `@파일명` 으로 첨부하면 AI가 해당 명세를 읽고 구현한다.

**예시**:
```
개발자로서 @docs/team-recruit-spec.md 를 구현해줘.
```

```
개발자로서 @docs/team-recruit-spec.md 중 "게시글 목록 조회" 기능만 먼저 구현해줘.
```

**출력물**: Java 소스 코드 (Controller / Service / Entity / Repository / DTO)

---

### 디자이너로 일 시키기

```
디자이너로서 @[기획서 파일명] 의 [화면명] 화면을 구현해줘.
```

**예시**:
```
디자이너로서 @docs/team-recruit-spec.md 의 "팀 모집 목록" 화면을 구현해줘.
```

```
디자이너로서 @docs/team-recruit-spec.md 의 모든 화면을 구현해줘.
```

**출력물**: `src/main/resources/templates/views/[도메인]/` 하위 HTML 파일

---

### 테스터로 일 시키기

```
테스터로서 [도메인명] 도메인의 코드 품질을 검토해줘.
```

```
테스터로서 [파일명] 이 하네스 규칙을 준수하는지 검토해줘.
```

```
테스터로서 [규칙 내용] 을 하네스에 추가하는 것을 제안해줘.  (Owner 승인 후 적용)
```

**출력물**: 품질 보고서 (위반 항목 목록 + 수정 방향)

## 전체 금지 목록 한눈에 보기

> 각 역할이 **절대 해선 안 되는 것**을 역할별로 모아놓은 참조 테이블.

### 기획자 금지

| # | 금지 내용 | 이유 |
|---|----------|------|
| 1 | 코드 작성 | 개발자 역할의 영역 |
| 2 | 구현 방법 지시 (어떤 기술을 쓸지 강제) | 개발자 자율성 침해 |
| 3 | 디자인 결정 (색상, 레이아웃) | 디자이너 역할의 영역 |
| 4 | 모호한 수락 조건 ("잘 동작해야 함") | 검증 불가능 |
| 5 | 하네스 파일 수정 | 테스터/Owner 전용 |

### 개발자 금지

| # | 금지 내용 | 이유 |
|---|----------|------|
| 1 | Entity에 `@Setter` 사용 | 불변성 위반, 의도 불명 상태 변경 |
| 2 | `FetchType.EAGER` 사용 | N+1 문제, 성능 저하 |
| 3 | Controller에서 Repository 직접 접근 | 레이어 의존 방향 위반 |
| 4 | Entity를 API 응답/View에 직접 반환 | 도메인 노출, 결합도 상승 |
| 5 | 기획서 요구사항 임의 변경·축소 | 기획자 역할의 결정사항 침해 |
| 6 | `System.out.println` 사용 | 운영 환경 로그 오염 |
| 7 | `@Lazy`로 순환 참조 회피 | 설계 문제 은폐 |
| 8 | 하네스 파일 수정 | 테스터/Owner 전용 |

### 디자이너 금지

| # | 금지 내용 | 이유 |
|---|----------|------|
| 1 | 인라인 스타일 (`style="..."`) 사용 | 유지보수 불가, 일관성 파괴 |
| 2 | 색상값 하드코딩 (`#3B82F6` 등) | CSS 변수 체계 파괴 |
| 3 | 아이콘 라이브러리 혼용 | 시각적 일관성 파괴 |
| 4 | 테이블에 `table-responsive` 누락 | 모바일 레이아웃 파괴 |
| 5 | 삭제 버튼에 확인 모달 없이 연결 | 실수 삭제 UX 문제 |
| 6 | Java/Service/Controller 코드 수정 | 개발자 역할의 영역 |
| 7 | 하네스 파일 수정 | 테스터/Owner 전용 |

### 테스터 금지

| # | 금지 내용 | 이유 |
|---|----------|------|
| 1 | 프로덕션 코드 직접 수정 | 발견·보고만 담당 |
| 2 | Owner 승인 없는 하네스 수정 | 하네스는 협의 사항 |
| 3 | 기능 추가·변경 제안 | 기획자 역할의 영역 |
| 4 | 위반 미해결 코드에 배포 허용 | 품질 게이트 역할 |

## 하네스 수정 방법 (테스터 + Owner 전용)

### 수정 대상 파일
```
.cursor/rules/
├── 00-global-direction.mdc   ← 전역 규칙
├── 01-planner-harness.mdc    ← 기획자 하네스
├── 02-developer-harness.mdc  ← 개발자 하네스
├── 03-designer-harness.mdc   ← 디자이너 하네스
└── 04-tester-harness.mdc     ← 테스터 하네스
.cursorrules                  ← 프로젝트 기술 규칙
```

### 수정 절차
1. **테스터가 개선 제안** → Owner에게 보고
2. **Owner 승인** 후 테스터가 해당 `.mdc` 파일 수정
3. **변경 내용 공유** → 전 팀원에게 변경 사항 전달

### AI에게 하네스 수정 요청

```
테스터로서 개발자 하네스에 다음 규칙을 추가해줘:
"금지: xxx는 하지 않는다. 이유: yyy"
```

## 기획서 파일 관리

기획서는 `docs/` 폴더에 보관한다.

```
docs/
├── community-spec.md       ← 커뮤니티 기능 기획서
├── team-recruit-spec.md    ← 팀 모집 기능 기획서
└── ...
```

기획서가 없으면 개발을 시작하지 않는다.
기획서 없이 개발 요청이 오면 기획자 역할로 먼저 기획서를 만들도록 유도한다.

## 자주 묻는 질문

**Q. 역할을 지정하지 않으면 어떻게 되나요?**  
A. 개발자 하네스가 기본으로 적용됩니다.

**Q. 기획서 없이 개발 요청을 해도 되나요?**  
A. 기획서 없이 시작하면 요구사항이 불명확해 재작업이 발생합니다. 반드시 기획자 역할로 기획서를 먼저 만드세요.

**Q. 하네스 규칙이 현실에 맞지 않아요.**  
A. 테스터에게 개선 제안을 요청하세요. 테스터가 Owner와 협의 후 수정합니다.

**Q. 디자이너가 백엔드 데이터 구조를 알아야 하나요?**  
A. 기획서의 "데이터 요구사항" 섹션을 참고하면 됩니다. 직접 Java 코드를 볼 필요는 없습니다.

**Q. 테스트 코드도 개발자 하네스를 따르나요?**  
A. 예. 테스트 코드 작성도 개발자 역할입니다. `src/test/` 하위 파일 작성 시 개발자 하네스를 따릅니다.
